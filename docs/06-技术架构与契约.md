# 06 — 技术架构与契约：实现细节

> 对应 `酒馆手机功能与实现方案.md` §8（技术方案）
> 本文描述技术栈、目录结构、App 注册契约、分层职责、生命周期和构建配置的工程实现约束。
>
> 仓库基线为 [Tavern Extension Template](https://github.com/puppywow/sillytavern-floating-phone)（Vite + Vue 3 + Pinia 插件模板）。模板已提供的基础设施标记为 `[模板]`，phone 插件新增/修改的标记为 `[新增]`。

---

## 1. 技术栈

### 1.1 模板已提供

| 技术 | 用途 |
|------|------|
| TypeScript | 主语言 |
| Vue 3 `<script setup>` | UI 框架 |
| Pinia | 状态管理 |
| zod 4 | Schema 校验 |
| klona | 安全克隆（去 Proxy 后写入 settings）|
| Tailwind CSS 4 | 样式框架（模板已安装但未启用，按需在 `global.css` 中 `@import`） |
| VueUse | Composition 工具集 |
| vue-final-modal | 模态框组件 |
| unplugin-auto-import | 自动导入 Vue/Pinia/zod 等 |
| unplugin-vue-components | 自动注册组件 |
| Vite 7 | 构建工具 |

### 1.2 Phone 插件需新增

| 技术 | 用途 |
|------|------|
| fflate | 浏览器端 ZIP 完整备份 |
| markdown-it + DOMPurify | Markdown 渲染（禁用原始 HTML 后 DOMPurify 净化）|
| vue-tsc | 类型检查（`pnpm typecheck`）|

### 1.3 强制依赖

- `window.TavernHelper` — 通过 `manifest.json` 声明 `dependencies: ["third-party/JS-Slash-Runner"]`，确保酒馆助手先于本插件加载
- 启动时仍须运行时检查 `window.TavernHelper` 实际可用性，不可用时阻断初始化

### 1.4 不引入

- 独立后端
- Node.js 浏览器不兼容模块
- 无 Tavern Helper 的降级生成路径

## 3. 分层职责

| 层 | 职责 | 不得 |
|----|------|------|
| **Component** | 展示 + 触发动作 | 不直接读写 SillyTavern API / 文件 / metadata |
| **Store** | UI 状态 + 业务编排 | 不承载全部解析细节 |
| **Service** | 楼层、生成、解析、剪贴板等无界面能力 | — |
| **Repository** | 持久化、迁移、导入导出、scope 切换 | — |
| **App 内部** | 该 App 专有页面、组件、Store | 不导入其他 App 内部实现 |

App 间仅通过公共服务、实体 ID 和导航参数协作。

## 4. App 扩展契约

### 4.1 注册定义

```ts
interface PhoneAppDefinition {
  id: string;                    // 稳定的小写 kebab-case
  name: string;
  icon: string;
  defaultRoute: string;
  defaultVisible: boolean;
  defaultDock: boolean;
  defaultOrder: number;
  load: () => Promise<PhoneAppModule>;  // 动态 import()
}
```

### 4.2 懒加载模块

```ts
interface AppRouteDefinition {
  path: string;                // 相对路径，如 ''、'detail/:id'
  component: Component;        // Vue 组件
  props?: boolean | Record<string, unknown>;
}

interface PhoneAppModule {
  routes: AppRouteDefinition[];
  generationAdapters?: GenerationAdapter<unknown, unknown>[];
  storageDomains?: StorageDomainDefinition[];
  dispose?: () => void | Promise<void>;
}
```

### 4.3 Storage Domain

```ts
interface StorageDomainDefinition {
  id: string;
  scope: 'chat' | 'global';
  schemaVersion: number;
  schema: z.ZodType<unknown>;
  createDefault: () => unknown;
  migrate: (data: unknown, fromVersion: number) => unknown;
}
```

### 4.4 注册约束

- 注册时校验 App ID、路由名、storage domain ID、generation action ID 全局唯一
- 内置 App 使用冻结的稳定 ID
- 新 ID 必须使用小写 ASCII kebab-case
- 文件名只能由已注册 ID 和内部 UUID 构造
- 未知旧 ID 保留在偏好数据中，不显示但不删除

### 4.5 新增 App 清单

新增 App 需要提供：
1. `PhoneAppDefinition` + `PhoneAppModule`（routes 必填）
2. 可选的 `storageDomains`（如需独立数据文件）
3. 可选的 `generationAdapters`（如需生成能力）
4. 无需修改手机壳、导航、生成内核或 storage index

## 5. 生命周期

### 5.1 初始化序列

```
1. manifest.json 加载 dist/index.js + dist/index.css
2. jQuery ready → 启动初始化
3. 等待 window.TavernHelper 就绪（运行时检查）
4. 依赖检查：TavernHelper 不可用 → 阻断并报错
5. 注册扩展菜单入口（"打开酒馆手机"）
6. 根据设置创建悬浮球
7. 监听 CHAT_CHANGED → resolveScopeBinding() → 检查恢复日志 → 加载数据
```

### 5.2 销毁序列

```
1. 停止活跃生成 → 等待楼层恢复
2. 卸载 Vue 应用
3. 移除根节点、动态样式节点
4. 销毁悬浮球
5. 执行 disposers 中全部解除函数
6. 模块引用置 null
```

### 5.3 状态机要点

- 初始化失败必须复位状态，允许重试
- 全局防重键 `window.__sillytavernPhonePluginState__`
- 热重载等待异步 `destroyPhone()` 完成后再重新初始化
- 页面卸载无法保证异步完成，依赖恢复日志在下次启动核对

## 6. 生成适配器契约

### 6.1 核心接口

```ts
interface GenerationAdapter<TConfig, TResult> {
  appId: string;
  actionId: string;
  configSchema: z.ZodType<TConfig>;
  buildRequest(config: TConfig): GenerationRequestParts;
  parse(raw: string, config: TConfig): ParseResult<TResult>;
  save(result: TResult, context: GenerationSaveContext): Promise<{ entityId: string }>;
}
```

### 6.2 buildRequest 约束

- 只能返回声明式区段和固定格式
- 不能直接调用 Tavern Helper
- 不能修改楼层状态
- 不能订阅全局事件

### 6.3 生成服务边界

- `GenerationService` 先校验 adapter/config
- 再执行 token 预检和来源隐藏事务
- `generate()` 返回后先恢复并核验楼层
- 之后才调用 adapter 的 `parse()`、预览和 `save()`
- 解析、预览和业务持久化不处于楼层隐藏窗口内

### 6.4 错误处理

- Adapter 抛出的校验、解析、保存错误统一转为结构化错误
- 保存失败不能重新发送生成请求
- 原始输出进入该 App 的 `FailedGenerationDraft`

## 7. 构建配置

### 7.1 入口与产物

- 单入口 `src/index.ts` → `dist/index.js`（ES module）+ `dist/index.css`
- `src/index.ts` 导入 `src/global.css`，Vite 合并 CSS 到 `dist/index.css`
- `manifest.json` 声明 `"js": "dist/index.js"` + `"css": "dist/index.css"`

### 7.2 emptyOutDir

模板使用 `emptyOutDir: false`。原因是 `dist/` 随仓库提交、CI 自动打包——`false` 避免本地构建清空远端产物造成冲突。Phone 插件开发期间可保持 `false`；发布独立构建时改为 `true`。

### 7.3 外部模块（不打包进 bundle）

- `@sillytavern/scripts/extensions` — extension_settings, getContext
- `@sillytavern/script` — eventSource, event_types, getRequestHeaders, saveChat, saveSettingsDebounced
- `@sillytavern/scripts/i18n` — t 翻译函数
- 浏览器全局：jQuery（`$`）、lodash（`_`）、showdown、toastr、Popper、hljs

### 7.4 打包进 bundle 的依赖

模板已提供：Vue、Pinia、zod、klona、VueUse、vue-final-modal

Phone 需新增：fflate、markdown-it、DOMPurify

### 7.5 构建命令

```bash
pnpm install        # 安装依赖
pnpm lint           # ESLint 检查
pnpm lint:fix       # ESLint 自动修复
pnpm typecheck      # [新增] vue-tsc 类型检查（须先安装 vue-tsc）
pnpm watch          # [模板] 开发模式持续构建（未压缩 + inline sourcemap）
pnpm build          # [模板] 生产构建（压缩 + 独立 sourcemap）
```

模板原生的 `pnpm build` 不含 typecheck 步骤。phone 插件应在 CI 或 `pnpm build` 前单独执行 `pnpm typecheck`。

### 7.6 TypeScript 配置补充

模板 `tsconfig.json` 需要补充以下项：

- `include` 添加 `@types/**/*.d.ts` 和 `src/**/*.vue`
- `exclude` 添加 `可参考拓展` 和 `docs`

### 7.7 ESLint 配置补充

模板 `eslint.config.mjs` 的 `globalIgnores` 需添加 `可参考拓展/**` 和 `docs/**`。

### 7.8 联调

- 开发期在仓库目录内独立 `pnpm watch`
- 实际联调将仓库放在 `SillyTavern/public/scripts/extensions/third-party/<插件目录>`
- `@sillytavern/*` 由 `vite.config.ts` 中的自定义 resolver 按插件安装后的固定层级解析
- `可参考拓展/` 和 `docs/` 不进入 TypeScript、Vite 和 ESLint 范围
