# 03 — 数据持久化：实现细节

> 对应 `酒馆手机功能与实现方案.md` §7（数据持久化）
> 本文描述存储架构、revision 文件、scope 绑定、导入导出、收藏快照和数据迁移的工程实现约束。

---

## 1. 存储架构

### 1.1 双层存储

| 层级 | 位置 | 用途 |
|------|------|------|
| 主存储 | `user/files/*.json` | 业务数据，SillyTavern 用户目录独立 JSON |
| 兜底存储 | `extension_settings.sillytavernPhone` | 读写失败时的备份，灾难恢复副本 |

- 优先读取独立文件；文件不存在、损坏或读写失败时回退 settings 兜底
- 文件可用时以文件内容为唯一活动数据源，不与兜底自动合并
- 正常关闭时刷新兜底；强制关闭允许兜底落后一个会话

### 1.2 SillyTavern 文件 API

| 操作 | 接口 | 说明 |
|------|------|------|
| 上传 | `POST /api/files/upload` | 需 `getRequestHeaders()` 认证 |
| 验证 | `POST /api/files/verify` | 检查文件是否存在 |
| 删除 | `POST /api/files/delete` | 删除前校验路径严格匹配 |
| 读取 | `GET /{path}` | `cache: 'no-store'` |

- 删除请求体 `{ path: 'user/files/...' }`，调用前校验路径
- storage index 读取使用 `cache: 'no-store'` + cache-busting 参数

### 1.3 数据分布

| 数据 | 文件 | 作用域 |
|------|------|--------|
| 活动文件指针和 scope 绑定 | `phone-storage-index.json` | 全局（固定文件名）|
| 日记 | `phone-{scopeId}-diary-{revisionId}.json` | 单聊天 |
| 论坛 | `phone-{scopeId}-forum-{revisionId}.json` | 单聊天 |
| 番外 | `phone-{scopeId}-extras-{revisionId}.json` | 单聊天 |
| 小剧场 | `phone-{scopeId}-theater-{revisionId}.json` | 单聊天 |
| 书信 | `phone-{scopeId}-letters-{revisionId}.json` | 单聊天 |
| 总结 | `phone-{scopeId}-summaries-{revisionId}.json` | 单聊天 |
| 收藏快照/偏好/外部API/提示词/八股规则/壁纸索引 | `phone-global-{domain}-{revisionId}.json` | 全局 |
| 自定义壁纸原文件 | `phone-wallpaper-{id}.{ext}` | 全局 |
| 恢复日志/操作日志/兜底数据 | `extension_settings.sillytavernPhone` | 全局 |

全局域 domain ID 枚举（`StorageDomainId`）：

| domain | 文件 | 说明 |
|--------|------|------|
| `favorite-snapshots` | `phone-global-favorite-snapshots-{revisionId}.json` | 跨聊天收藏只读快照 |
| `preferences` | `phone-global-preferences-{revisionId}.json` | UI 偏好（主题/悬浮球/主屏布局/阅读器） |
| `text-api-settings` | `phone-global-text-api-settings-{revisionId}.json` | 外部文本 API 配置 |
| `prompt-settings` | `phone-global-prompt-settings-{revisionId}.json` | App/类型提示词 + 快速短语 |
| `bagu-rules` | `phone-global-bagu-rules-{revisionId}.json` | 八股检测规则 |
| `wallpaper-index` | `phone-global-wallpaper-index-{revisionId}.json` | 壁纸索引 |

以上 domain ID 由内置 App 注册表登记，后续 App 可通过 `StorageDomainDefinition` 扩展。

## 2. Revision 文件机制

### 2.1 不可变修订

除 `phone-storage-index.json` 外，所有 JSON 数据文件使用不可变命名（含 `revisionId`）：

```
保存流程：
1. 将新修订写入新文件（含新 revisionId）
2. 回读校验（文件内容与写入一致）
3. 一次性更新 storage index 指针
4. 提交成功后删除旧修订文件
5. 删除旧修订失败 → 记录为待清理，不影响已提交数据
```

### 2.2 Storage Index

```ts
interface PhoneStorageIndex {
  schemaVersion: number;
  revision: string;
  scopeBindings: Record<string, ScopeBinding>;
  scopeFiles: Record<string, Partial<Record<StorageDomainId, string>>>;
  globalFiles: Partial<Record<StorageDomainId, string>>;
  updatedAt: string;
}
```

- Storage index 是全部 revision 的唯一入口
- 每次提交后立即镜像到 `extension_settings.sillytavernPhone`
- Index 镜像不受 `5 MiB` 配额淘汰
- 主 index 损坏时用镜像恢复指针
- Index 与镜像都不可用时停止自动清理，仅允许导入完整备份

### 2.3 并发写入

- 写入按逻辑键（`scopeId + domainId` 或全局 domain）建立串行队列
- 所有 storage index 提交经过全局 commit mutex
- 提交前重新读取最新 index，只替换目标指针
- 避免两个不同领域并发保存时互相覆盖

### 2.4 操作日志

写新修订前先把目标路径写入 settings 操作日志。页面若在索引提交前崩溃，下次启动可删除未被 index 引用的暂存修订。

## 3. Scope 绑定

### 3.1 scopeId 创建

```
首次进入聊天 →
  1. getContext() 取得 chatMetadata
  2. 写入 chatMetadata.sillytavernPhone.scopeId（新 UUID）
  3. saveMetadata() 持久化
  4. phone-storage-index.json 记录 scopeBindings[scopeId]
```

### 3.2 ownerKey

```ts
interface ScopeBinding {
  ownerKey: string;    // 单角色: char:{avatarId}, 群聊: group:{groupId}
  chatId: string;
  displayName: string;
  updatedAt: string;
}
```

- `ownerKey` 使用角色卡头像文件 ID（`char:{avatarId}`）或群聊 ID（`group:{groupId}`）
- 不使用易变的 `characterId/this_chid` 作为身份主键
- 监听 `CHARACTER_RENAMED(old_avatar, new_avatar)`，同步更新 ownerKey

### 3.3 聊天切换判定

```
CHAT_CHANGED → resolveScopeBinding():

1. metadata 无 scopeId，或 scopeId 已不在 storage index
   → 按 ownerKey + chatId 查找唯一 binding
   → 命中 → 补写 scopeId（惰性修复）
   → 无命中 → 创建新 scopeId
   → 多个命中 → 暂停让用户选择

2. scopeBindings[scopeId].ownerKey + chatId 匹配当前聊天
   → 正常沿用

3. chatId 不同 + main_chat 指向原聊天（Branch/Checkpoint）
   → 创建新 scopeId + 复制全部 chat-scoped domain
   → 复制收藏快照到新 scopeId
   → 更新 metadata 与 binding

4. chatId 不同 + 旧聊天不存在（改名）
   → 保留 scopeId，更新 binding

5. 无法判定 → 暂停让用户选择
```

### 3.4 Scope 事务

涉及 metadata 和 storage index 两个保存位置时，采用可恢复步骤：

1. 在 settings 操作日志登记新 scopeId 和目标修订
2. 写入并校验全部 chat-scoped domain 及收藏快照修订
3. 提交 storage index
4. 写聊天 metadata
5. 清除操作日志

任一步失败保留旧状态，下次启动根据日志继续或回退。

## 4. 数据原则

### 4.1 Schema 版本

- 所有持久数据带 `schemaVersion`
- 低于当前版本 → 逐版本迁移链升级
- 高于插件支持版本 → 拒绝写入并提示升级插件
- 不存在旧版聊天变量、localStorage 的迁移需求

### 4.2 ID 与时间

- 所有 ID 使用 UUID，时间使用 ISO 8601
- 正文统一命名为 `content`
- 章节号、回复层级、书信方向、来源楼层均保存结构化字段

### 4.3 写入约束

- 写入前移除 Vue Proxy（`toRaw()`）
- 主文件和 settings 兜底均失败时必须记录并提示
- 只重写发生变化的 `scopeId + 内容类型` 文件

### 4.4 删除

- 删除分册时同步清理关联的收藏快照
- 删除未绑定 scope 时：先删文件 API → 成功或 404 → 提交 index 更新
- 删除壁纸：先删原文件 → 成功或 404 → 提交索引

### 4.5 清空当前聊天

- 写入空 revision → 移除该 scope 全部收藏快照 → 一次 index 提交
- 不删除 scope 绑定
- 旧文件在提交后清理

### 4.6 未知 domain 保护

- 当前版本不认识的 domain：保留指针、不加载、不迁移、不自动删除
- 完整备份原样包含
- 只能在数据恢复页显示并手动处理

## 5. 导入导出

### 5.1 完整备份（ZIP）

```
phone-backup-{timestamp}.zip
├─ manifest.json          # 版本、schemaVersion、文件清单、SHA-256
├─ phone-storage-index.json
├─ phone-{scopeId}-diary-{revisionId}.json
├─ phone-global-favorite-snapshots-{revisionId}.json
├─ phone-wallpaper-{id}.{ext}
└─ ...
```

- `manifest.json` 记录每个条目的相对路径、字节数、MIME、SHA-256
- 导入先核对清单 → 大小 → 哈希，任一失败拒绝整个备份
- 全量预检 + 二次确认后一次 index 提交
- 压缩包上限 `100 MiB`，解压后 `200 MiB`
- 拒绝绝对路径、`..`、重复条目、未声明文件
- API Key 默认脱敏

### 5.2 选择性导入（JSON）

- 只能选择内容类型，合并到当前 `scopeId`
- 导入实体及子实体统一生成新 UUID
- 重映射分册、回复父子、章节总结等内部引用
- 上限 `50 MiB`
- 保留导入实体的收藏状态，创建新快照

### 5.3 不进入备份的内容

- 来源隐藏恢复日志
- scope/import 操作日志
- `pendingFavoriteSyncs`
- 旧 revision 清理队列
- settings 兜底副本

这些属于本机瞬时状态，只能写入可选诊断报告。

### 5.4 安全约束

- 导入对象递归拒绝 `__proto__`、`prototype`、`constructor` 等危险键
- 动态字典使用 `Map` 或 `Object.create(null)`
- 不把未验证对象直接 `Object.assign` 到 Store

## 6. 收藏快照

### 6.1 快照结构

```ts
interface GlobalFavoriteSnapshot {
  key: string;                    // `${sourceScopeId}:${contentType}:${entityId}`
  sourceScopeId: string;
  contentType: ContentType;
  entityId: string;
  title: string;
  updatedAt: string;
  payload: FavoriteSnapshotPayload;
}

// 六类创作内容的只读详情 DTO（zod discriminated union）
type FavoriteSnapshotPayload =
  | { type: 'diary'; item: DiaryEntry }
  | { type: 'forum'; item: ForumThread }
  | { type: 'extras'; item: ExtraChapter }
  | { type: 'theater'; item: TheaterEntry }
  | { type: 'letters'; item: LetterEntry }
  | { type: 'summaries'; item: SummaryEntry };

// 同步操作日志条目
interface PendingFavoriteSync {
  key: string;
  operation: 'upsert' | 'delete';
  expectedEntityUpdatedAt?: string;
  createdAt: string;
}
```

### 6.2 跟随原文语义

- 原文编辑 → 同步更新快照
- 原文取消收藏 → 同步删除快照
- 原文删除 → 同步删除快照
- 分册删除 → 同步清理其中实体的快照

### 6.3 事务安全

- 编辑、收藏或删除时：同时写好实体 revision 与快照 revision → 一次 index 提交同时切换两个指针
- 提交前失败 → 旧版本同时有效
- `pendingFavoriteSyncs` 记录暂存文件和预期操作，供崩溃后重试
- 待删除快照立即从 UI 隐藏，不因重试而短暂复活

## 7. Settings 兜底

### 7.1 容量管理

- Settings 兜底总序列化上限 `5 MiB`
- 恢复日志、scope pending 日志和小型全局配置优先保留
- 超限时停止加入大体积创作快照，明确列出未进入兜底的文件
- 正常关闭时只刷新本次变更的领域，不重写未变化数据

### 7.2 聊天级兜底

- 只保留最近活动的一个 `scopeId`
- 切换 scope 后替换旧兜底，不累积所有聊天全文

### 7.3 启动恢复

- 启动时主文件可用 → 以文件为准
- 主文件不可用 → 回退 settings 兜底
- Index 与镜像都不可用 → 阻止自动恢复，只允许导入备份
