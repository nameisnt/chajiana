## 1. 产品定位

酒馆手机是运行在 SillyTavern 页面中的手机式创作工具箱，以 SillyTavern 第三方扩展插件形式安装和加载。

入口和使用路径：

```text
扩展菜单中的“打开酒馆手机”入口 / 可选悬浮球
  → 插件在 SillyTavern 主页面挂载手机弹窗
  → 手机主屏
  → 打开创作、管理或工具 App
```

插件只负责组织聊天素材、发起文本生成、解析结果和保存创作内容。默认生成通道下，以下能力由当前 SillyTavern 环境负责：

- 当前角色卡与聊天；
- 当前文本 API、模型和采样参数；
- 当前预设；
- 当前世界书；
- 酒馆原生聊天消息。

手机内不再实现角色卡、预设、世界书、生图、相册、角色画像或世界观管理。文本生成默认使用酒馆当前 API，也允许用户在手机设置 App 中配置一个可选的外部文本 API；无论选择哪种文本 API，角色卡、预设和世界书仍使用酒馆当前环境。

## 2. 当前功能基线

### 2.1 手机框架

- 桌面端约 `360 × 700`，移动端约 `92vw × 85vh`。
- 支持亮色/暗色主题、毛玻璃卡片、手机圆角外壳和底部 Home 条。
- 状态栏显示时间、灵动岛、信号、Wi-Fi、电量和关闭按钮。
- 支持扩展菜单入口打开、关闭后再次打开；插件不在 SillyTavern 扩展设置页提供独立设置面板。
- 悬浮球默认关闭，可在设置中开启，并可设置尺寸和颜色。
- 主屏以分页网格和 Dock 展示 App。
- App 可排序、隐藏或加入 Dock，主屏布局持久化。
- 页面导航使用栈语义：打开 App、进入子页、替换当前页、返回、回主屏。
- 从收藏进入详情后，返回时应回到原收藏列表和筛选状态。

## 3. 通用生成能力

### 3.1 生成流程

所有创作 App 共用一条生成管线：

```text
选择来源楼层
  → 可选引用手机内已有创作内容
  → 快照全部楼层的原始隐藏状态并写入恢复日志
  → 临时隐藏来源范围以外的可见楼层
  → 将 App/类型提示词、引用素材、额外要求和固定格式要求组合为用户输入
  → 调用 Tavern Helper generate()
  → 当前 SillyTavern 自动应用角色卡、当前预设和当前世界书，并按所选通道使用酒馆当前文本 API 或外部文本 API
  → 无论成功、停止或失败，都恢复全部楼层的原始隐藏状态
  → 解析约定 XML
  → 预览、直接保存或流式展示
  → 保存到对应内容库
```

生成请求必须遵守以下边界：

- 只调用 `generate()`，不传 `preset_name`、`injects`、`ordered_prompts` 或
  `overrides.chat_history`，直接使用酒馆当前预设、世界书和原生聊天历史构建流程。默认通道不传
  `custom_api`；选择外部文本 API 时只额外传入 `custom_api`。
- 来源范围以外的原本可见楼层只在生成事务期间临时设为隐藏；生成结束后当前聊天的隐藏状态必须与生成前完全一致。
- 使用 `should_silence: true`，避免手机生成任务占用或污染酒馆正常回复流程。
- 每次生成设置唯一 `generation_id`；流式事件按该 ID 过滤，停止时调用 `stopGenerationById()`。

禁止重新引入 `generateRaw()`、自建
`ordered_prompts`、手机内预设选择、提示词注入、手机内世界书拼接、永久隐藏来源外楼层或修改楼层正文。外部文本 API 不得绕过 Tavern
Helper 直接 `fetch('/v1/chat/completions')`，避免另造一套提示词与聊天历史管线。

### 3.2 来源楼层

来源楼层表示“本次内容写的是哪些楼层”，支持：

- 最新楼层；
- 从 0 至指定楼层；
- 全部楼层；
- 指定单层；
- 最近 N 楼；
- 自定义范围。

具体读取规则：

- 读取范围后只保留可见楼层，每楼只取当前选中的消息页，不发送隐藏楼层或未选中的 swipe。
- 若范围解析后没有任何原本可见的来源楼层，则阻止生成并提示重新选择，不能隐藏全部聊天后使用空历史继续生成。
- 不重新组装
  `{ role, content }`，不改写正文，也不附加楼层号或发言者标签；由酒馆原生流程保留群聊角色、当前 swipe、正则和世界书处理。
- 生成前快照所有楼层的
  `message_id + is_hidden`；只把“原本可见且不在来源集合内”的楼层临时设为隐藏，原本隐藏的楼层始终保持隐藏。
- 若来源集合已经覆盖全部原本可见楼层，则直接生成，不创建恢复日志，也不执行无意义的隐藏写入。
- 隐藏和恢复统一使用 `setChatMessages(..., { refresh: 'none' })`
  并等待 Promise 完成，避免页面楼层闪烁；楼层仍处于临时隐藏状态时不得调用 `saveChat()`，只修改当前内存中的聊天状态。
- 恢复必须写在 `finally`
  中，生成成功、解析失败、用户停止、API 异常都执行；恢复时按快照逐层还原，不能简单地把所有楼层取消隐藏。确认仍是原聊天且恢复后的身份哈希与快照匹配后调用一次
  `saveChat()`，只把已经恢复原状的聊天状态落盘。
- 整个事务绑定启动时的 `scopeId` 和任务 ID；恢复前必须再次核对当前聊天 metadata 中的
  `scopeId`，禁止把旧作用域的楼层号应用到其他作用域。
- 同一时间只允许一个来源隐藏事务；启动前若酒馆或手机已有生成任务则拒绝开始。事务期间由 `SendGuard`
  拦截用户点击酒馆发送按钮和在聊天输入框按 Enter 发送，但不调用全局
  `deactivateSendButtons()`/`activateSendButtons()`，避免错误改变其他插件或酒馆自己的按钮状态。
- `SendGuard`
  只添加带插件命名空间的捕获阶段监听和提示遮罩；若仍检测到酒馆或其他扩展开始了新的非本任务生成，立即停止手机任务并先恢复楼层。事务结束只移除自己创建的监听和遮罩，不负责“重新启用”酒馆按钮。
- 若生成期间发生聊天切换，立即停止对应生成，不在新聊天执行恢复；保留恢复日志，并在原聊天再次激活时优先恢复。

崩溃恢复日志按 `scopeId` 保存在
`extension_settings.sillytavernPhone.pendingVisibilityRecoveries`，不同聊天的未恢复事务不能互相覆盖：

```ts
interface PendingVisibilityRecovery {
  scopeId: string;
  generationId: string;
  createdAt: string;
  messages: Array<{
    messageId: number;
    isHidden: boolean;
    role: 'system' | 'assistant' | 'user';
    name: string;
    contentHash: string;
  }>;
}

type PendingVisibilityRecoveries = Record<string, PendingVisibilityRecovery>;
```

- 必须先持久化并等待恢复日志保存成功，才能隐藏任何楼层。
- `contentHash` 使用 Web Crypto SHA-256 计算
  `role + name + 当前 swipe 正文`，不使用易碰撞的手写整数哈希；恢复日志本身若超过 settings 可保存上限或保存失败，则阻止本次生成，不能在没有恢复凭据时继续隐藏。
- 正常恢复、重新读取确认当前状态并成功 `saveChat()` 后才清除日志；如果身份哈希不匹配则不得保存，保留日志进入人工恢复。
- 插件启动和每次聊天切换后都先取得当前 `scopeId`，只恢复 `pendingVisibilityRecoveries[currentScopeId]`；`chatId`
  不参与恢复匹配，避免聊天改名后无法恢复。
- 自动恢复前用 `role + name + contentHash`
  核对楼层身份；匹配的原楼层按快照恢复，新增加的楼层保持现状，已删除的楼层跳过并记录。
- 任一现存 `messageId`
  的身份哈希不符，说明楼层曾被删除、重排或替换；此时停止自动修改、保留日志，并在手机设置 App 的“数据恢复”页面提供逐项核对和人工恢复，禁止盲目按楼层号写回。
- 日志超过合理时限仍不得直接丢弃，只能在确认目标聊天不存在或由用户明确放弃后清理。

2026-06-23 在当前酒馆对最后一楼进行了两组临时隐藏测试并恢复原状：只调用 `setChatMessages(..., { refresh: 'none' })`
时，当前内存立即变为隐藏，但切换聊天再回来后恢复原文件状态；额外调用 `saveChat()`
后，隐藏状态会写入聊天文件并在重开后继续存在。因此来源事务明确禁止在临时隐藏阶段调用 `saveChat()`，但在 `finally`
已恢复并核验后必须保存一次。恢复日志仍保留，因为生成期间 SillyTavern 或其他扩展可能触发保存；若异常重载后实际楼层已是原状态，恢复器核对并保存确认后清除日志。

所有保存结果应记录可读来源文本和可排序的来源楼层值。

### 3.3 引用素材

- 可引用手机内已经保存的日记、论坛、番外、小剧场、书信和总结。
- 引用选择器按内容类型和分册分组。
- 支持搜索、展开/收起、分组全选和取消。
- 引用素材放入本次请求的 `user_input` 参考资料区，不伪装成聊天楼层。
- 引用只参与本次用户输入，不复制为新的持久数据关系。

续写和回应类生成仍必须重新选择来源楼层，不能只靠手机内旧内容直接生成。各动作追加到 `user_input` 的既有内容固定如下：

- 番外续写：选中的来源楼层 + 当前分册在续写点之前的全部章节。启用书级总结时，总结只替换它已经覆盖的较早章节，未被总结覆盖的章节仍发送原文。
- 论坛回复：选中的来源楼层 + 当前帖子的主楼、现有普通回复和楼中楼关系。
- 书信回信：选中的来源楼层 + 用户在生成界面选择的最近 N 封相关往返信件，按时间顺序发送。
- 日记“让他人阅读”：选中的来源楼层 + 当前被阅读的这一篇日记；不自动附带同一本日记中的其他条目。
- 这些旧内容属于“引用素材及额外上下文”区，位于 App/类型提示词之后、用户额外要求之前，不伪装成聊天楼层。

### 3.4 输出模式

- 结果去向 `resultMode` 与传输方式 `stream` 是两个独立配置，不把流式传输作为第三种结果模式：

```ts
interface GenerationOutputConfig {
  resultMode: 'preview' | 'save';
  stream: boolean;
}
```

- `resultMode: 'preview'`：生成并解析后进入预览，再确认保存。
- `resultMode: 'save'`：解析成功后直接保存并进入刚保存内容的详情页。
- `stream: true`：生成期间显示实时文本；结束后仍按 `resultMode` 进入预览或直接保存。
- `stream: false`：等待完整结果返回后再解析，并按 `resultMode` 继续。
- 生成期间可停止；界面必须明确区分生成中、停止、失败和完成。
- 用户停止后不得按 `resultMode: save`
  自动保存业务实体；已有流式文本作为未完成原文进入解析修复/失败草稿，用户可编辑后手动重新解析，没有可用文本时只记录停止状态。

### 3.5 预览与解析修复

- 预览页可修改标题、正文、类型、时间和来源等适用字段。
- XML 缺失或格式异常时展示解析警告。
- 保留模型原始返回，允许用户编辑原文后重新解析。
- 解析失败不能丢失原文；允许作为失败草稿继续编辑。
- 保存后必须进入新内容详情，而不是回到空配置页。

### 3.6 提示词

- App 级提示词：日记、论坛、番外、小剧场、书信、总结。
- 特殊提示词：日记阅读反应、番外章节总结。
- 类型级提示词：番外和小剧场。
- 类型提示词支持 `<char>`、`{{char}}`、`<user>`、`{{user}}` 替换。
- `user_input`
  按固定顺序组合：App/类型创作提示词 → 引用素材及额外上下文 → 用户额外要求 → 固定输出格式；空的可选区段直接省略。
- App、特殊和类型提示词允许编辑；XML/结构化输出格式由各 App 的解析器契约固定维护，不允许用户编辑，并始终追加在
  `user_input` 末尾。
- 批量生成由任务队列拆成多个“单请求、单 `<result>`”任务，逐项完成来源事务、解析和暂存；不要求模型在一次响应中返回多个
  `<result>`。停止批量任务只阻止未开始项并停止当前 `generation_id`，已经成功保存/暂存的项不回滚。
- 支持预览、复制、重置、导出和导入。
- 快速短语按分组管理，可一键追加到“额外要求”。

### 3.7 文本生成通道

生成配置提供两种文本通道，默认使用酒馆当前 API：

```ts
type TextGenerationProvider =
  | { mode: 'tavern' }
  | {
      mode: 'external';
      apiUrl: string;
      apiKey?: string;
      model: string;
      contextWindow?: number;
      maxOutputTokens?: number;
    };
```

- `tavern`：调用 `generate()` 且不传 `custom_api`，使用酒馆当前文本 API、模型和采样参数。
- `external`：第一版只支持 OpenAI-compatible 反向代理，仍调用同一个 `generate()`，仅通过
  `custom_api: { apiurl, key, model, source: 'openai', max_tokens? }` 切换文本 API；`max_tokens` 只有用户填写
  `maxOutputTokens` 时才覆盖，否则沿用当前预设。当前预设、世界书、角色卡、临时可见的来源楼层以及 `user_input`
  组合规则保持不变。API Key 允许为空，以兼容本地无鉴权服务。
- 配置和测试流程同时参考 `docs/reference/pyq-creator-main` 与 `docs/reference/sillytavernDIARY-main`：提供启用开关、API
  URL、遮罩的 API Key、模型选择、保存、测试连接和刷新模型；不照搬前者的 `localStorage`/浏览器直连，也不照搬后者的
  `/gen`、全局事件劫持和强制关闭流式传输。
- URL 保存和请求前统一执行 `normalizeExternalApiUrl()`：去除首尾空白、末尾 `/`、末尾 `/models` 和末尾
  `/chat/completions`，持久层只保存规范化后的基址。
- URL 必须能由 `new URL()` 解析且协议仅允许 `http:`/`https:`；拒绝 URL 内嵌用户名密码以及
  `file:`、`javascript:`、`data:` 等协议。API Key 只能进入独立字段，不能拼进 URL、查询串或日志。
- 连接测试与模型刷新调用 Tavern Helper `getModelList({ apiurl, key })`；该接口和 Diary 一样通过 SillyTavern
  `/api/backends/chat-completions/status`
  代理请求，不让浏览器直接访问外部域名。模型 ID 去空、去重并排序；原模型仍存在则保留，否则选择返回列表第一项。
- 外部 API 设置使用 storage index 中的 `text-api-settings` 全局逻辑文件，实际保存为带 revision 的 JSON；字段为
  `enabled/url/key/model/models/contextWindow/maxOutputTokens/updatedAt/lastTestedAt`，并沿用 settings 兜底。URL 或 Key 改变后立即清空旧模型列表和
  `lastTestedAt`，必须重新测试；只改变已验证列表内的模型不要求重测。模型接口通常不返回上下文长度，因此这两个 token 字段允许用户留空并回退酒馆当前值。
- 与 Diary 的文件存储一样，API
  Key 在外部 API 设置修订文件和失败兜底中只能以明文保存，插件没有可用的系统密钥库，不能宣称已加密；设置页必须明确提示这一点。界面默认遮罩，不写日志、不进入错误提示正文，普通数据导出默认将
  `key` 替换为空字符串；只有用户明确选择“包含 API 密钥”并确认风险后才导出原值。
- 测试、保存和生成开始时都读取一份不可变配置快照；生成途中修改设置只影响下一个任务。外部 API 的 `stream`
  仍遵循统一生成配置，不因切换通道被悄悄关闭，也不在流式失败后自动重发同一请求。
- 外部 API 未配置、连接测试失败或所选模型不存在时阻止该通道生成，用户可以随时切回酒馆当前 API。

## 4. 各 App 详细功能

### 4.1 日记

页面流：


### 4.4 小剧场

- 类型页支持搜索类型名和历史内容。
- 类型按使用频率排序。
- 支持添加自定义类型。
- 配置项包括参与角色、类型提示词、角色/用户替换和渲染模式。
- 渲染模式支持 Markdown 文本和 iframe 前端渲染。
- 用户在生成前选择渲染模式；Markdown 模式要求 `<content>` 中返回文本/Markdown，前端渲染模式要求 `<content>`
  中返回 HTML。两种模式共用同一个外层输出字段，不在生成后猜测内容类型。
- 解析器把最外层 `<content>`
  起止标签之间的内容作为正文原样提取：Markdown 模式按文本保存，前端渲染模式按 HTML 保存。HTML 不需要另加 CDATA 或另一套 JSON/XML 协议；固定格式只要求正文自身不得输出会提前闭合外层字段的字面量
  `</content>`。
- 前端渲染在结构上参考 Tavern
  Helper 的“把完整前端内容放入独立 iframe”方案，但渲染对象是手机持久层保存的正文，不是酒馆聊天楼层；不得为了渲染而创建临时楼层，也不得调用依赖
  `message_id` 的 `formatAsDisplayedMessage()`。
- 同时参考 `可参考拓展/st-theater-main/index.js`
  的完整 HTML 提示词、`iframe.srcdoc`、原始 HTML 独立缓存和响应式高度范围；只借鉴这些渲染职责，不照搬它的独立生成/API/预设/世界书、SillyTavern 设置面板、IndexedDB 主存储或 HTML 猜测提取流程。
- 默认 frontend 类型提示词沿用该参考项目“完整可独立运行、响应式、默认不引用外部资源、直接输出 HTML 而非 Markdown 代码块”的约束；手机解析仍以本方案固定
  `<result><content>` 契约为准，不用正则猜测代码块、`<html>`、`<snow>` 或纯文本，也不擅自补全被截断的模型输出。
- 手机使用共享 `FrontendFrame` 组件，把保存的原始 HTML 经 `buildFrontendDocument()` 补齐为完整文档后写入
  `iframe.srcdoc`。HTML 片段需要自动补充 `doctype`、`meta viewport` 和基础页面样式；完整 HTML 文档则保留其
  `head`、`style`、`script` 和 `body` 内容。
- iframe 负责与手机壳及 SillyTavern 主页面隔离样式和脚本，并在详情内容区内部滚动。固定使用
  `sandbox="allow-scripts"`，不开放 `allow-same-origin`、顶层导航、表单提交、下载、弹窗或父页面 DOM 权限。
- `st-theater-main` 为读取 `contentDocument.scrollHeight` 使用了
  `allow-scripts allow-same-origin`；本插件明确不采用。`buildFrontendDocument()` 注入只上报内容高度的内部脚本，通过
  `postMessage` 把有限数值发给父组件；父组件校验 `event.source === iframe.contentWindow`、实例 channel
  ID、消息类型和数值范围后，在手机内容区允许的最小/最大高度间 clamp。高度消息没有导航、存储或执行命令能力，失败时回退为固定内部滚动高度。
- 前端渲染采用平衡型 CSP：允许 HTTPS 图片、字体、音视频、脚本和样式以及必要的 `data:`/`blob:` 静态资源，但禁止
  `fetch`、XHR、WebSocket、EventSource、嵌套页面、插件对象和表单外发。`buildFrontendDocument()`
  必须注入以下策略，正文不能放宽：

```text
default-src 'none';
img-src https: data: blob:;
media-src https: data: blob:;
font-src https: data:;
style-src 'unsafe-inline' https:;
script-src 'unsafe-inline' https:;
connect-src 'none';
frame-src 'none';
object-src 'none';
base-uri 'none';
form-action 'none';
navigate-to 'none';
```

- 此模式直接渲染已保存的 HTML，不自动执行聊天楼层专属的酒馆宏、酒馆正则或 Markdown 格式化；类型提示词和固定输出格式必须要求模型返回可独立运行的 HTML。若以后需要宏替换，应在构造
  `srcdoc` 前增加独立、可测试的文本预处理步骤，不能借用聊天楼层。
- 进入其他详情、切换渲染模式、关闭手机、删除内容或销毁手机时卸载 iframe，并清理其事件监听和临时 URL；普通关闭只保留导航栈、详情 ID 和表单状态，不保留正在运行的 iframe。再次打开手机时由原始正文重新构造
  `srcdoc`。持久层只保存原始正文和渲染模式，不保存 iframe DOM 或运行后的页面状态。
- 支持普通生成、预览、保存、复制、收藏、编辑和删除。
- 提供可搜索的生成历史。

### 4.5 收藏

- 只显示已收藏内容，覆盖日记、论坛、番外、小剧场、书信和总结。
- 支持按内容类型筛选。
- 支持按来源角色和聊天筛选。
- 当前聊天作用域保存原始内容；收藏时同步插件用户级只读快照，用于跨聊天查看。
- 当前聊天中的收藏可进入原详情页。
- 其他聊天的收藏打开只读快照，不能错误跳转到当前聊天的同 ID 内容。
- 跨聊天快照采用“跟随原文”语义：收藏中的原文被编辑时同步更新快照；取消收藏、删除原文或删除所属分册时同步删除快照，不保留永久档案。
- 快照使用 `scopeId + 内容类型 + entityId`
  组成稳定唯一键，同一原文只能有一个快照。编辑、收藏或删除时先同时写好实体 revision 与快照 revision，再用一次 storage
  index 提交同时切换两个指针；提交前失败时旧版本同时有效。`pendingFavoriteSyncs`
  记录暂存文件和预期操作，供崩溃清理/重试；待删除快照立即从 UI 隐藏，不能因重试而短暂复活。
- 支持管理模式、批量取消收藏和批量删除。

- 可删除整本书信。

### 4.7 总结

### 4.8 提示词工坊

### 4.10 阅读聊天

页面流：

```text
当前角色卡全部聊天目录 → 聊天阅读页 → AI 楼层标题与正文
```

- 使用 Tavern Helper `getChatHistoryBrief('current')`
  获取当前角色卡的聊天文件目录，再把选中的单个 brief 项包装成数组传给
  `getChatHistoryDetail([briefItem])`，按需读取该聊天的完整内容。
- 2026-06-23 在当前酒馆对三个真实聊天实测：详情返回结构为 `Record<文件名, 消息数组>`；正文位于 `mes`，AI 消息满足
  `is_user === false` 且不是 narrator/system 类型，`swipe_id` 表示当前页而 `mes` 已是当前选中正文，`swipes`
  保存全部消息页，`is_system` 对应 Tavern Helper `ChatMessage.is_hidden` 使用的原始隐藏标记。实现时先经过
  `normalizeArchivedMessage()` 转成内部强类型，业务组件不得直接使用接口的 `any`。
- 只读取并展示 AI 输出楼层，不展示用户输入、narrator/system 消息或其他非 AI 楼层；每楼只使用 `mes`，不展开未选中的
  `swipes`。
- 默认排除原聊天中已隐藏的 AI 楼层；设置中提供“显示隐藏 AI 楼层”开关，开启后在目录和详情中明确标记隐藏状态。
- brief 数据只稳定提供文件名、文件大小、总消息数等摘要，不保证直接提供 AI 楼层数量。目录首次显示文件名、更新时间/文件信息和总楼层数；某聊天详情首次读取后再缓存并显示准确 AI 楼层数量，禁止为了首屏计数一次性加载当前角色卡的全部聊天全文。
- 每个 AI 楼层转换为 `{ title, body, messageId }` 阅读项；默认标题为“第 N 楼”，默认正文为该楼原始 AI 输出。
- 用户可设置标题和正文两组正则替换规则，每组包含查找表达式、替换字符串和 flags：

```ts
interface ChatReaderRegexRule {
  find: string;
  replace: string;
  flags: string;
}

interface ChatReaderSettings {
  title: ChatReaderRegexRule;
  body: ChatReaderRegexRule;
  showHiddenAssistantMessages: boolean;
}
```

- 标题规则对单楼 AI 原文执行替换后得到标题；正文规则对同一原文独立执行替换后得到正文，两组规则互不串联。
- 规则为空时使用默认值；正则无匹配时标题回退到“第 N 楼”、正文回退到原文；非法正则不得保存，并在界面显示错误位置。
- flags 只允许不重复的 `gimsuy`；正则替换在独立 Web Worker 中对当前选中聊天整批执行，单次任务超过 `1500ms`
  即终止 Worker、保留原文并提示规则可能发生灾难性回溯，不能让主页面线程卡死。
- 单楼替换结果设置 `2 MiB`
  上限，超限视为规则失败；失败时整批不提交部分结果。恢复默认、修改规则或刷新聊天都会创建新 Worker，旧任务结果不得覆盖新规则。
- Worker 的 Blob
  URL 在任务完成、超时、规则变化和组件卸载时都要 terminate/revoke；长聊天阅读列表使用虚拟滚动，不能一次把全部正文节点挂进手机 DOM。
- 正则设置保存在 storage index 的 `preferences`
  全局逻辑文件中，切换聊天或角色卡后继续使用；聊天正文只实时读取，不复制到手机数据文件。归档详情只在当前阅读页内缓存，离开详情、手动刷新或收到
  `CHAT_CHANGED` 后立即丢弃；每次重新打开详情都再次调用
  `getChatHistoryDetail([briefItem])`，不依赖文件大小判断同长度编辑。
- 目录提供显式刷新按钮；刷新会重新请求 brief、清除本角色的详情缓存和已计算 AI 楼层数。这样不批量下载全部聊天全文，也不会让长期 session 缓存掩盖外部修改。
- 切换角色卡后清空目录与正文缓存并重新读取；本功能范围为当前单角色卡的全部聊天，群聊归档暂不纳入。

## 5. 数据与持久化

### 5.1 存储边界

存储网关参照 `docs/reference/sillytavernDIARY-main/src/storage/file-storage-api.js`
的文件读写与兜底方式：大块数据优先保存为 SillyTavern 用户目录下的独立 JSON 文件，并以 `extension_settings`
作为读写失败兜底。只参考这一存储机制，不沿用 Diary 按角色名归类数据的方式。

| 数据                                                                                                                | 存储位置                                                 | 作用域                          |
| ------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- | ------------------------------- |
| 活动文件指针和 scope 绑定                                                                                           | `user/files/phone-storage-index.json`                    | 当前 SillyTavern 用户全局提交点 |
| 日记                                                                                                                | `user/files/phone-{scopeId}-diary-{revisionId}.json`     | 单个聊天 scope                  |
| 论坛                                                                                                                | `user/files/phone-{scopeId}-forum-{revisionId}.json`     | 单个聊天 scope                  |
| 番外                                                                                                                | `user/files/phone-{scopeId}-extras-{revisionId}.json`    | 单个聊天 scope                  |
| 小剧场                                                                                                              | `user/files/phone-{scopeId}-theater-{revisionId}.json`   | 单个聊天 scope                  |
| 书信                                                                                                                | `user/files/phone-{scopeId}-letters-{revisionId}.json`   | 单个聊天 scope                  |
| 总结                                                                                                                | `user/files/phone-{scopeId}-summaries-{revisionId}.json` | 单个聊天 scope                  |
| 跨聊天收藏、偏好、外部 API、提示词、八股规则和壁纸索引                                                              | `user/files/phone-global-{domain}-{revisionId}.json`     | 当前 SillyTavern 用户全局       |
| 自定义壁纸原文件                                                                                                    | `user/files/phone-wallpaper-{wallpaperId}.{ext}`         | 当前 SillyTavern 用户全局       |
| 插件版本、启用状态、来源楼层恢复日志、`pendingStorageOperations`、`pendingFavoriteSyncs` 和各 JSON 文件失败兜底数据 | `extension_settings.sillytavernPhone`                    | 当前 SillyTavern 用户全局       |

所有 revision 数据文件使用统一包装格式；固定 storage index 使用紧随其后的独立结构：

```ts
interface PhoneStorageEnvelope<T> {
  schemaVersion: number;
  kind: string;
  updatedAt: string;
  data: T;
}

interface ScopeBinding {
  ownerKey: string;
  chatId: string;
  displayName: string;
  updatedAt: string;
}

interface PhoneStorageIndex {
  schemaVersion: number;
  revision: string;
  scopeBindings: Record<string, ScopeBinding>;
  scopeFiles: Record<string, Partial<Record<StorageDomainId, string>>>;
  globalFiles: Partial<Record<StorageDomainId, string>>;
  updatedAt: string;
}
```

六类既定创作内容仍使用冻结的 `ContentType`，但 storage index 的键使用可注册
`StorageDomainId`，从而允许后续 App 增加独立 chat/global domain，而不修改生成内核或把新数据硬塞进现有六类文件。

- 文件通过 SillyTavern `/api/files/upload`、`/api/files/verify`、`/api/files/delete` 和 `user/files/...`
  读取接口访问，请求头使用 `getRequestHeaders()`。删除请求使用
  `{ path: 'user/files/...' }`，调用前必须校验路径严格匹配本插件登记的文件名，禁止接受任意用户输入路径。
- 读取固定的 storage index 必须使用 `cache: 'no-store'`
  和 cache-busting 查询参数；revision 文件名天然不可变，回读校验仍使用 `no-store`，不能让浏览器缓存返回上一版索引。
- 除固定的 `phone-storage-index.json` 外，所有 JSON 数据文件都是带 `revisionId`
  的不可变修订。保存时先把新修订写入新文件并回读校验，再一次性更新 storage
  index 指针；索引提交失败时旧指针和旧数据仍然有效。
- 更新 storage
  index 成功后再删除旧修订；删除失败只记录为待清理文件，不影响已提交数据。写新修订前先把目标路径写入 settings 操作日志，页面若在索引提交前崩溃，下次启动可删除未被 index 引用的暂存修订。
- 启动时优先读取独立文件；文件不存在、格式损坏或读写失败时，回退到 `extension_settings.sillytavernPhone`
  中对应的兜底快照。
- `phone-storage-index.json` 是所有修订的入口，必须在每次成功提交后把完整 index 镜像到 settings，且该镜像不受创作内容
  `5 MiB`
  配额淘汰；主 index 损坏时先用镜像恢复活动指针。index 与镜像都不可用时停止自动清理文件，只允许用户导入完整备份，禁止猜测最新 revision。
- 文件可用时以对应文件内容为唯一活动数据源，不与 settings 中的兜底副本自动合并；每次只重写发生变化的
  `scopeId + 内容类型` 文件，不得因修改一篇日记而重写其他 scope 或其他五类创作内容。
- 正常关闭手机弹窗时，将本次打开期间发生变化的各领域最新快照写入 `extension_settings.sillytavernPhone`
  兜底；未变化的领域不重复写入。聊天级兜底只保留最近活动的一个
  `scopeId`，切换 scope 后替换旧 scope 兜底，不在 settings 中累计所有聊天全文。
- settings 兜底设置总序列化上限 `5 MiB`；来源恢复日志、scope
  pending 日志和小型全局配置优先保留，超限时停止加入大体积创作快照并明确列出未进入兜底的文件。完整备份包才是长期全量备份。
- 导入、迁移、清空等大范围操作完成后立即刷新对应 settings 兜底，不等待关闭手机。
- 页面崩溃或被强制结束可能来不及执行关闭备份，因此 settings 兜底允许落后于主文件一个打开会话；它是灾难恢复副本，不是第二个实时主仓库。
- 写入按逻辑键（`scopeId + StorageDomainId`
  或全局 domain）建立串行队列，而不是按每次都不同的 revision 文件名排队；所有 storage index 提交再经过一个全局 commit
  mutex，并基于提交前重新读取的最新 index 只替换目标指针，避免两个不同领域并发保存时互相覆盖索引更新。
- 产品按单个 SillyTavern 页面会话设计，不承诺同一用户同时打开多个标签页或设备写入时的数据合并；这种场景采用最后写入者覆盖，并在文档中明确不支持。
- 清理浏览器缓存、Cookie 或 `localStorage` 不会删除这些数据；插件提供完整备份包以防 SillyTavern 用户目录被删除或迁移。

聊天级数据通过稳定作用域定位：

- 首次进入聊天时，通过 `getContext()` 取得 `chatMetadata`，写入 `chatMetadata.sillytavernPhone.scopeId` 后等待
  `saveMetadata()` 完成；当前类型包没有
  `updateChatMetadata()`，不得把不存在的辅助函数写进实现。聊天正文和创作数据不写入 metadata。
- `phone-storage-index.json` 使用 `scopeBindings[scopeId]` 记录稳定
  `ownerKey`、`chatId`、显示名和更新时间，活动文件指针统一保存在
  `scopeFiles[scopeId]`；六类既定创作正文及后续 App 的 chat-scoped domain 都只存在各自修订文件中。
- `ownerKey` 对单角色聊天使用角色卡头像文件 ID（`char:{avatarId}`），对群聊使用
  `group:{groupId}`；易受角色列表排序影响的 `characterId/this_chid` 只可作为运行时信息，不能持久化为身份主键。
- 监听 `CHARACTER_RENAMED(old_avatar, new_avatar)`，在一个 storage index 提交中把相关 binding 的 `ownerKey`
  从旧头像 ID 更新为新头像 ID；更新失败时保留旧值和 pending 日志，不能因角色改名把全部聊天误判为未绑定。
- 聊天改名后沿用原 `scopeId`，不能按角色名或聊天名重新建库；角色同名、不同聊天和群聊之间不得串数据。
- SillyTavern 创建 Branch/Checkpoint 时会复制原聊天 metadata，因此新聊天会暂时携带相同 `scopeId`。每次 `CHAT_CHANGED`
  后必须先执行 `resolveScopeBinding()`，完成 scope 判定后才能检查隐藏恢复日志或加载业务数据：

```text
metadata 没有 scopeId，或 metadata 中的 scopeId 已不在 storage index
  → 先按 ownerKey + chatId 查找唯一 binding
  → 唯一命中则补写该 scopeId（用于完整备份导入后的惰性修复）
  → 没有命中则创建新 scopeId 和空数据
  → 多个命中则暂停并让用户选择，禁止猜测

scopeBindings[scopeId].ownerKey 和 chatId 都等于当前聊天
  → 正常沿用

chatId 不同，且 metadata.main_chat 指向原聊天或已绑定的旧聊天仍存在
  → 判定为分支/复制
  → 创建新 scopeId
  → 逐个复制原 scope 的全部 chat-scoped domain；已注册 domain 经 schema 深拷贝，未知 domain 按不可变原始 revision 保留
  → 为复制后仍处于收藏状态的实体建立以新 scopeId 为键的新快照
  → 更新当前聊天 metadata 与新绑定

chatId 不同，且旧聊天已不存在
  → 判定为聊天改名
  → 保留 scopeId，仅更新 scopeBindings

仍无法判定
  → 暂停加载并让用户选择“沿用原作用域”或“复制为独立作用域”
```

- scope 绑定同时涉及聊天 metadata 和 `phone-storage-index.json`
  两个保存位置，无法依赖一次写入同时成功。实现采用可恢复步骤：先在 settings 操作日志登记新 scopeId 和目标修订，写入并校验全部已注册 chat-scoped
  domain 及收藏快照修订，再提交 storage
  index，最后写聊天 metadata 并清除日志。任何一步失败都可按旧 index 继续使用原数据；下次启动根据日志继续完成或回退，未完成前不能加载目标 scope 或进入来源隐藏事务。这就是本文所说的“scope 事务”，不是数据库事务。

2026-06-23 对当前角色卡已有的三个聊天进行了两轮实际切换测试并恢复到原聊天：主聊天的 `main_chat` 为空，`Branch #27` 的
`main_chat` 指向主聊天，`Branch #12` 的 `main_chat` 指向 `Branch #27`，能够形成直接父链；`Branch #27` 与由它产生的
`Branch #12` 还具有相同的一组扩展 metadata 键，符合分支复制 metadata 的行为。三个聊天的 `integrity` 与 `chat_id_hash`
完全相同，因此这两个字段只能作为同源辅助信息，不能作为聊天或 scope 的唯一 ID。本插件仍需把自己的 UUID `scopeId`
写入 metadata，并以 `chatId + main_chat + scopeBindings` 完成判定。

### 5.2 数据原则

- 所有持久数据必须带 `schemaVersion`，通过显式迁移升级。
- 低于当前版本的数据只按逐版本迁移链升级；高于插件支持版本的数据拒绝写入并提示升级插件，禁止按默认值强行降级解析。
- 新 Store 不直接散落调用存储 API，统一经过 Repository/Gateway。
- 写入文件前移除 Vue Proxy；主文件和 settings 兜底均失败时必须记录并提示。
- 聊天切换时重新加载聊天级数据，不能覆盖新聊天。
- ID 必须在数据作用域内稳定，不能只依靠数组索引。
- 删除分册时同步清理其中收藏实体对应的全局快照。
- 编辑、取消收藏和删除实体时通过 `favoriteSnapshots`
  Repository 同步更新或删除全局快照；失败操作进入可重试日志，业务 App 不直接写快照文件。
- 导入必须先完整校验和迁移，再按“完整备份覆盖全部、选择性导入合并当前 scope”的规则一次性提交；任何文件失败时保留导入前数据，不允许半导入。
- 不存在旧版聊天变量、旧 `localStorage` 或旧插件数据的迁移要求；只维护本插件自身 `schemaVersion` 之间的升级。
- 自定义壁纸不以 base64 塞进 JSON；索引只保存文件名、MIME、尺寸和显示设置，图片保存为独立原文件。壁纸文件失败时明确报错，不写入聊天 metadata 或
  `extension_settings` 兜底。
- 壁纸上传限制单文件 `10 MiB`、最大
  `8192 × 8192`，同时校验 MIME、文件扩展名和实际图片解码结果；拒绝 SVG 和伪装成图片的 HTML。
- 壁纸预览使用的 Object URL 在更换、删除和组件卸载时必须 `URL.revokeObjectURL()`；正式显示使用已上传的 `user/files`
  URL，不把完整 base64 常驻 Store。
- 完整导出使用 ZIP 备份包而不是单个 JSON：包内包含 `manifest.json`、全部逻辑数据 JSON 和壁纸原文件；API
  Key 默认脱敏。选择性内容导出仍可使用单个 JSON。来源隐藏恢复日志、scope/import 操作日志、`pendingFavoriteSyncs`、旧 revision 清理队列和 settings 兜底副本属于本机瞬时状态，只能写入可选诊断报告，不能作为可导入数据重新激活。导入不得覆盖未成功解析和迁移的数据。
- `manifest.json`
  记录插件版本、各 domain 的 schemaVersion、每个条目的相对路径、字节数、MIME 和 SHA-256；导入在 zod 解析前先核对清单、大小和哈希，任何缺失、多余或校验失败都拒绝整个备份包。
- 删除聊天不会自动删除对应 scope；设置 App 的数据恢复页按需扫描聊天绑定，列出未绑定 scope，用户可先导出再手动删除，插件不得静默清理。
- 用户确认删除未绑定 scope 后，按 storage index 中登记的活动修订和待清理修订逐一调用
  `/api/files/delete`；404 视为已不存在，其他失败保留 index 记录并允许重试，全部活动指针移除并成功提交 index 后才删除 scope 绑定。删除壁纸同样先删除原文件，成功或 404 后再提交索引更新。
- “清空当前聊天创作数据”不删除 scope 绑定：先对六类既定内容及用户确认包含的后续已注册 chat-scoped
  domain 写入空 revision，并同时写入移除该 scope 全部收藏快照后的全局 revision，再用一次 storage
  index 提交切换；旧文件在提交后清理，失败时仍显示清空前数据。
- 当前版本不认识的 domain 不参与普通清空、迁移或自动清理，只能在数据恢复页显示路径、大小和来源版本，并由用户导出后明确删除。

导入策略固定为：

- 完整备份导入：先在内存中解包并用 zod 校验全部 manifest、JSON 和壁纸，再展示将被替换的 scope、全局设置和文件数量。用户二次确认后，把所有数据写成新的 revision 文件并回读校验，最后只提交一次
  `phone-storage-index.json`
  切换全部活动指针；提交前崩溃仍使用旧 index，提交后旧修订延迟清理，不依赖逐文件覆盖或内存回滚。
- 选择性导入：只能选择内容类型并合并到当前
  `scopeId`，导入实体及其全部子实体统一生成新 UUID，同时重映射分册、回复父子关系、章节总结覆盖关系等内部引用，绝不按原 ID 覆盖当前内容。
- 选择性导入保留实体的收藏状态；提交成功后为导入的收藏实体创建对应的新全局快照。
- 完整备份默认不含 API Key；备份中没有 `key` 时保留本机现有 Key，只有备份明确包含 Key 且用户在导入页再次确认时才替换。
- 完整导入保留备份中的 `scopeId` 和 binding；当前打开聊天立即按 `ownerKey + chatId`
  修正 metadata，其他聊天在以后首次打开时惰性核对并补写 scopeId。找不到对应聊天的导入 scope 保留为未绑定数据，不按聊天名猜测归属。
- ZIP 导入只接受 manifest 声明的相对路径，拒绝绝对路径、`..`、重复条目和未声明文件；压缩包上限 `100 MiB`、解压后总量上限
  `200 MiB`、单项和条目数也必须受限，防止 zip bomb。选择性 JSON 上限 `50 MiB`，所有限制在解包/解析前检查。
- 导入对象递归拒绝 `__proto__`、`prototype`、`constructor` 等危险键；动态字典使用 `Map` 或
  `Object.create(null)`，不能把未验证对象直接 `Object.assign` 到 Store/设置，防止原型污染。

### 5.4 插件数据契约

本插件不需要复刻任何旧字段。实现时直接以以下契约建立 zod Schema；所有 ID 使用 UUID，所有时间使用 ISO
8601 字符串，正文统一命名为 `content`：

```ts
type ContentType = 'diary' | 'forum' | 'extras' | 'theater' | 'letters' | 'summaries';
type StorageDomainId = string; // 运行时必须由内置配置或 App 注册表登记，不能接受任意用户输入
type RenderMode = 'markdown' | 'frontend';

interface SourceSelection {
  scopeId: string;
  chatIdAtGeneration: string;
  mode: 'latest' | 'fromStart' | 'all' | 'single' | 'recent' | 'range';
  ranges: Array<{ start: number; end: number }>;
  messageIds: number[];
  label: string;
  sortKey: number;
}

interface ContentBase {
  id: string;
  title: string;
  content: string;
  source: SourceSelection;
  favorite: boolean;
  createdAt: string;
  updatedAt: string;
}

interface FailedGenerationDraft {
  id: string;
  contentType: ContentType;
  rawOutput: string;
  warnings: string[];
  source: SourceSelection;
  createdAt: string;
}

interface GlobalFavoriteSnapshot {
  key: string; // `${sourceScopeId}:${contentType}:${entityId}`
  sourceScopeId: string;
  contentType: ContentType;
  entityId: string;
  title: string;
  updatedAt: string;
  payload: FavoriteSnapshotPayload; // 六类只读详情 DTO 的 zod discriminated union
}

type FavoriteSnapshotPayload =
  | { type: 'diary'; item: DiaryEntry }
  | { type: 'forum'; item: ForumThread }
  | { type: 'extras'; item: ExtraChapter }
  | { type: 'theater'; item: TheaterEntry }
  | { type: 'letters'; item: LetterEntry }
  | { type: 'summaries'; item: SummaryEntry };

interface PendingFavoriteSync {
  key: string;
  operation: 'upsert' | 'delete';
  expectedEntityUpdatedAt?: string;
  createdAt: string;
}
```

- `SourceSelection.messageIds` 保存生成当时实际参与的原本可见楼层，范围标签只用于展示，后续不得重新解析标签来推断来源。
- 成功实体不长期重复保存完整模型原文；只有解析失败草稿保存 `rawOutput`，成功保存后删除对应失败草稿。
- 收藏状态保存在原实体的 `favorite` 字段；当前 scope 的收藏列表由六类文件实时派生，不另建容易失步的 `FavoriteRef`
  主数据。

ID，但解析和重新保存时保留未知 ID，并把新增 App 按默认顺序补入，不能因版本新增、暂时下架或降级破坏用户布局。

### 5.5 固定生成输出契约

固定格式由解析器维护，并始终放在 `user_input` 末尾。所有动作最外层统一使用 `<result>`，正文统一使用
`<content>`；用户在小剧场生成前选择的渲染模式只改变 `<content>` 内部要求。

```xml
<!-- 日记、番外章节、小剧场、总结的基础形态 -->
<result>
  <title>标题</title>
  <content>正文；小剧场 frontend 模式时这里直接放 HTML</content>
</result>

<!-- 论坛首帖；后续回复只返回 replies -->
<result>
  <board>板块名</board>
  <title>帖子标题</title>
  <author>作者</author>
  <content>主楼正文</content>
  <replies>
    <reply><author>作者</author><content>回复正文</content><parent_ref></parent_ref></reply>
  </replies>
</result>

<!-- 书信 -->
<result>
  <title>标题</title>
  <content>正文</content>
</result>
```

- 番外章节号、书信格式、发信者、收信者、参与者和来源范围以生成请求中的结构化配置为准，不要求模型重复输出，也不允许模型覆盖用户已经选择的配置。
- 论坛生成前为已有回复分配临时引用 `E1/E2/...`，本次新回复按顺序为 `N1/N2/...`；`parent_ref` 可为空、引用已有 `E*`
  或更早的新回复 `N*`。保存时立即转换为真实
  `parentReplyId`；向后引用、自引用、循环或不存在的引用降级为普通回复并产生解析警告。
- 小剧场 frontend 模式以最外层 `<content>` 起止标签做边界提取并保存内部原文；Markdown 模式提取同一字段后按文本处理。
- 解析器允许外围解释文字并提取第一个完整 `<result>`；存在多个
  `<result>`、缺少必需字段或正文为空时进入解析修复，不擅自选择或保存。

### 6.3 分层职责

- 组件只负责展示和触发动作，不直接读写 SillyTavern 文件、metadata 或 extension settings。
- Store 负责当前 UI 状态和业务编排，不承载所有解析细节。
- Service 负责楼层、生成、解析、剪贴板等无界面能力。
- Repository 负责持久化、迁移、导入导出和作用域切换。
- App 间只能通过公共服务、实体 ID 和导航参数协作，避免互相导入组件内部状态。

#### 6.3.1 App 模块与后续扩展契约

每个 App 是独立模块，由注册表发现和懒加载；手机壳、导航、生成内核和存储网关不得反向导入某个 App 的页面、Store、Parser 或 Repository。内置 App 使用本文已经冻结的稳定 ID，后续 App 必须分配新的稳定 ID，发布后不能改名或复用。

```ts
type AppId = string;

interface PhoneAppDefinition {
  id: AppId;
  name: string;
  icon: string;
  defaultRoute: string;
  defaultVisible: boolean;
  defaultDock: boolean;
  defaultOrder: number;
  load: () => Promise<PhoneAppModule>;
}

interface PhoneAppModule {
  routes: AppRouteDefinition[];
  generationAdapters?: GenerationAdapter<unknown, unknown>[];
  storageDomains?: StorageDomainDefinition[];
  dispose?: () => void | Promise<void>;
}

interface StorageDomainDefinition {
  id: StorageDomainId;
  scope: 'chat' | 'global';
  schemaVersion: number;
  schema: z.ZodType<unknown>;
  createDefault: () => unknown;
  migrate: (data: unknown, fromVersion: number) => unknown;
}
```

- `appRegistry` 只保存轻量定义；进入 App 时才执行动态 `import()`，首次加载失败显示独立错误，不拖垮手机壳和其他 App。
- 注册时校验 App ID、路由名、storage domain ID 和生成 action ID 全局唯一；冲突直接阻止该 App 注册。
- 新 ID 只能使用稳定的小写 ASCII kebab-case；内置 App 保留现有 ID，后续 domain/action 以 App
  ID 为命名空间。所有文件名只能由已注册 ID 和内部 UUID 构造，不能直接采用导入数据或用户输入。
- 主屏布局按稳定 App ID 保存。新 App 首次出现时按 `defaultOrder/defaultDock/defaultVisible`
  合并到现有布局；未知旧 ID 不显示，但保留在原始偏好数据中，避免临时下架后用户布局丢失。
- `phone-storage-index.json`
  对已注册 domain 提供读写；遇到当前版本不认识的 domain 指针时只保留、不加载、不迁移、不自动删除。完整备份必须原样包含未知 domain，防止插件降级或 App 暂时移除造成数据丢失。
- 每个 App 只能通过注入的导航、生成、存储、对话框和剪贴板公共接口工作；不得直接访问其他 App
  Store，也不得自行挂主页面节点或注册第二套全局生命周期。
- 当前只保证仓库内后续 App 按此契约扩展。任意第三方运行时代码注入、远程 App 商店和跨插件 API 版本协商不在第一版范围内。

### 6.4 生命周期

- `manifest.json` 声明
  `display_name`、`loading_order`、`js: "dist/index.js"`、`css: "dist/index.css"`、版本、作者、主页、空的 `requires`
  和 Tavern Helper `dependencies`；`src/index.ts` 的构建结果在 SillyTavern `APP_READY` 后执行一次初始化。
- 收到 `APP_READY` 后等待 `window.TavernHelper`
  完成初始化，再创建运行时和读取数据；依赖缺失或初始化失败时显示阻断错误，不使用原生接口拼装另一套降级生成实现。
- 在 SillyTavern 扩展菜单注册一个最小“打开酒馆手机”入口，按需创建手机，不重复挂载 Vue；该入口只负责启动，不承载任何设置项。
- 悬浮球直接挂载在 SillyTavern 主页面，设置变化后即时同步。
- 提供显式 `destroy()`，在插件热重载、禁用或页面卸载时清理监听器、计时器、Vue 实例和页面节点。
- 监听聊天切换，刷新聊天级 Repository；必要时安全关闭当前详情页。
- 所有 `eventSource.on`、DOM 和窗口事件监听都必须保存对应解除函数并在销毁时停止。

### 6.5 悬浮球与手机弹窗实现

悬浮球和手机弹窗属于插件运行层，不属于任何 App。两者都直接挂载到 SillyTavern 主页面，且每个页面只能存在一个实例。

实现结构以 `docs/reference/sillytavernDIARY-main` 的插件版悬浮窗和对话框为参考基线：

- 参考 `float-window-manager.js` 的主页面 `body` 挂载、`eventsBound`/`windowAttached`
  防重、拖拽与点击区分、位置恢复和越界校正。
- 参考各 dialog manager 的“首次追加到 `body`，以后只显隐、不重复创建”模式。
- 参考 `plugin-initializer.js` 的全局初始化状态，热重载或重复入口不能产生第二套 UI 和监听器。
- 参考 `theme-manager.js` 的唯一样式节点 ID；重新加载样式前先移除旧节点，避免主题 CSS 累积。
- 不照搬 Diary 的放射子菜单和 jQuery 鼠标/触摸双事件；本插件悬浮球只负责打开手机，并统一用 Pointer Events 实现拖拽。

#### 6.5.1 运行时状态

建议由 `core/phoneRuntime.ts` 统一维护：

```ts
interface PhoneRuntime {
  app: App<Element> | null;
  appBridge: { suspendFrontendFrames(): void } | null;
  root: HTMLElement | null;
  dynamicStyleNodes: Array<HTMLStyleElement | HTMLLinkElement>;
  floatRoot: HTMLElement | null;
  floatButton: HTMLButtonElement | null;
  floatStyle: HTMLStyleElement | null;
  phoneVisible: boolean;
  disposers: Array<() => void>;
}
```

运行时另设全局防重键 `window.__sillytavernPhonePluginState__`，至少记录 `initializing`、`initialized`
和当前实例；初始化失败必须复位 `initializing`，允许用户重试。

状态转换：

```text
插件加载并收到 APP_READY
  ├─ 在扩展菜单注册“打开酒馆手机”入口
  └─ 若设置已启用 → 创建悬浮球

扩展菜单入口 / 点击悬浮球
  → 首次调用时创建 Vue 根节点和手机弹窗
  → 显示弹窗
  → 隐藏悬浮球

关闭按钮 / 点击遮罩
  → 隐藏弹窗但不卸载 Vue
  → 若设置仍启用则恢复悬浮球

插件热重载、禁用 / 页面卸载
  → 卸载 Vue
  → 删除根节点、样式、悬浮球
  → 解除全部事件监听和观察器
```

“关闭”与“销毁”必须分开：普通关闭保留当前 App、导航栈和未提交表单，但必须先卸载所有内容渲染 iframe；插件禁用、热重载或页面卸载才彻底销毁整个 Vue 应用。

#### 6.5.2 主页面挂载

插件宿主、手机根节点和悬浮球运行在 SillyTavern 主页面上下文，不依赖宿主 iframe 或
`window.parent`；只有小剧场等内容详情按需创建内部渲染 iframe：

- 创建节点前确认 `document.body` 和 `document.head` 已存在。
- 手机根节点直接追加到 `document.body`。
- 基础 `dist/index.css` 由插件 manifest 固定加载，不使用 `teleportStyle()`；主题等运行时样式使用带唯一 ID 的
  `<style>/<link>` 节点。
- 所有 DOM ID 和 class 使用 `phone-creative-` 或 `pc-` 前缀，避免污染 SillyTavern。
- 如果根节点被其他扩展移除，下次打开时检测 `isConnected` 并安全重建。
- 手机根节点、悬浮球根节点和动态样式分别使用固定唯一 ID；创建前先复用有效节点或移除失效残留。

#### 6.5.3 插件启动入口

初始化时只向 SillyTavern 扩展菜单追加一个带插件专属 ID 的“打开酒馆手机”入口，不加载 `settings.html`，不向
`#extensions_settings2` 追加配置面板；主题、悬浮球、存储、生成和阅读等设置全部在手机的“设置”App 内完成。

- 重复加载不能重复追加启动入口或注册监听。
- DOM 监听的解除函数必须加入 `disposers`。
- 点击菜单入口和点击悬浮球调用同一个 `showPhone()`/`togglePhone()`，禁止维护两套打开逻辑。

#### 6.5.4 手机弹窗挂载

首次显示时执行：

1. 使用 `document.createElement('div')` 创建插件专属根节点。
2. 设置固定 ID `phone-creative-root` 和 `aria-label="酒馆手机创作助手"`。
3. 将根节点追加到 `document.body`。
4. 创建独立 Pinia 实例并挂载根组件：`createApp(App).use(createPinia())`。
5. 通过 provide/inject 向根组件提供 `hidePhone()`，关闭按钮不直接依赖运行时模块。
6. 确认插件样式已由 manifest 入口或构建产物加载到主页面。

根节点和弹窗结构：

```html
<div id="phone-creative-root">
  <div class="phone-root" data-theme="light|dark">
    <div class="phone-modal">
      <div class="phone-modal__content">
        <section class="phone-shell">...</section>
      </div>
    </div>
  </div>
</div>
```

弹窗样式要求：

```css
.phone-modal {

  display: flex;
  align-items: center;
  justify-content: center;

  background: rgb(0 0 0 / 42%);
  pointer-events: auto;
}

.phone-shell {
  width: 360px;
  height: 700px;

}

@media (max-width: 768px) {

  }

    width: 92vw;

    max-width: 360px;
    max-height: 700px;
    border-radius: 28px;
  }

```

- 遮罩点击调用 `hidePhone()`。
- 手机内容区阻止点击冒泡，操作手机内部不能关闭弹窗。
- 状态栏关闭按钮也调用 `hidePhone()`。
- `Escape` 键关闭弹窗，但输入法组合状态下不得误触发。
- 弹窗显示后将焦点移到手机壳或第一个可操作元素；关闭后尽量把焦点还给打开入口。
- 手机壳可保留左边缘右滑返回：起点距左侧不超过 `24px`，水平移动至少 `72px`，垂直漂移不超过
  `42px`；表单控件和按钮上不触发。

#### 6.5.5 悬浮球 DOM 与层级

悬浮球单独挂载到 SillyTavern 主页面，不放在 Vue 根节点内。这样手机尚未创建时也能打开，手机隐藏时也不依赖 Vue 存活。

```html
<div class="pc-float-ball-root">

```

层级和交互要求：

```css
.pc-float-ball-root {

  inset: 0 auto auto 0;
  z-index: 2147482000; /* 低于手机弹窗 */
  pointer-events: none;
}

.pc-float-ball {
```

- 默认尺寸 `44px`，允许范围 `28–80px`。
- 默认颜色 `#007aff`。
- 根容器不拦截页面操作，只有按钮本身接收指针事件。
- 悬浮球层级低于手机遮罩，手机打开时仍应主动隐藏，避免读屏或键盘访问到不可见入口。
- 使用真实 `button`，而不是只有点击事件的 `div`。

#### 6.5.6 悬浮球拖拽

统一使用 Pointer Events，同时支持鼠标、触摸和触控笔：

1. `pointerdown` 记录起点和按钮当前位置，并对当前 pointer 调用 `setPointerCapture()`。
2. 桌面鼠标/触控笔移动超过 `5px`、触摸移动超过 `15px` 后才进入拖拽状态，沿用参考插件区分点击和拖拽的阈值思路。
3. `pointermove` 根据位移更新 `left/top`。
4. 位置始终限制在可视区域内，与四边至少保持 `8px`。
5. `pointerup`/`pointercancel` 释放捕获并保存最终位置。
6. 只有本次手势没有进入拖拽状态时，`click` 才打开手机。

边界计算：

```ts
const maxX = Math.max(8, viewportWidth - ballSize - 8);
const maxY = Math.max(8, viewportHeight - ballSize - 8);
const x = clamp(nextX, 8, maxX);
const y = clamp(nextY, 8, maxY);
```

- 初次无保存位置时，默认放在右下区域，并避开页面底部约 `100px`。
- 触摸结束后保留约 `300ms` 的防误触窗口，避免拖拽结束立即触发打开。
- 保存 `floatBallX`、`floatBallY` 到 `preferences` 全局逻辑文件的 UI 偏好。
- 页面缩放、旋转或 resize 后重新 clamp，防止球留在屏幕外。
- 恢复位置时若尺寸尚不可测，按默认球尺寸计算；首次布局完成后再次校正。
- 位置保存失败不影响本次拖拽。

#### 6.5.7 设置同步

UI 设置由 storage index 指向的 `preferences` 全局逻辑文件持久化，至少包含：

```ts
interface FloatBallSettings {
  floatingBallEnabled: boolean;
  floatingBallSize: number;
  floatingBallColor: string;
  floatBallX?: number;
  floatBallY?: number;
}
```

本实现优先采用事件驱动同步：

- 设置 Store 修改后调用运行时 `applyFloatBallSettings()`。
- 启用时立即创建，关闭时立即删除。
- 尺寸或颜色变化时只更新 CSS 变量，不重复创建按钮。
- 主窗口的 `focus`、`pageshow`、`resize`、`orientationchange`、`visibilitychange` 用于恢复和校正。
- 可使用 `MutationObserver` 检测主页面意外移除根节点。
- 只有在宿主页面兼容性确有需要时才增加低频轮询兜底，不把固定轮询作为主要同步机制。

#### 6.5.8 显隐与销毁 API

```ts
function showPhone(): void {
  ensurePhone();
  runtime.root?.removeAttribute('hidden');
  runtime.phoneVisible = true;
  setFloatBallVisible(false);
}

async function hidePhone(): Promise<boolean> {
  if (generationStore.hasActiveTask) {
    const confirmed = await confirmStopAndClose();
    if (!confirmed) return false;
    await generationStore.stopActiveTask();
    const restored = await sourceVisibility.waitForRestore();
    if (!restored) {
      openRecoveryBlockingView();
      return false;
    }
  }
  runtime.appBridge?.suspendFrontendFrames();
  runtime.root?.setAttribute('hidden', '');
  runtime.phoneVisible = false;
  setFloatBallVisible(settings.floatingBallEnabled);
  void enqueueFallbackFlushForChangedDomains();
  return true;
}

async function destroyPhone(): Promise<void> {
  await generationStore.stopAndRestoreIfNeeded();
  runtime.app?.unmount();
  runtime.root?.remove();
  runtime.dynamicStyleNodes.splice(0).forEach(node => node.remove());
  destroyFloatBall();
  runtime.disposers.splice(0).forEach(dispose => dispose());
}
```

实现时注意：

- `ensurePhone()` 必须幂等。
- 手机已显示时再次触发入口可选择保持显示或执行 toggle，但行为必须统一。
- 禁用悬浮球后关闭手机，不得把悬浮球错误恢复出来。
- 遮罩、关闭按钮和 Escape 都调用同一个异步 `hidePhone()`；存在生成任务时必须确认停止，并等待
  `stopGenerationById()`、来源恢复、恢复后的 `saveChat()` 全部完成后才隐藏。恢复失败时保持手机可见并进入阻断式恢复页。
- `suspendFrontendFrames()` 只卸载 iframe 实例，不清空导航栈、当前详情 ID 或未提交表单；下次 `showPhone()`
  根据原始正文重建。
- 每次 `hidePhone()`
  只把本次打开期间发生变化的领域加入同一条兜底刷新队列；刷新失败必须记录并在下次打开时提示，不能产生未处理的 Promise
  rejection。
- 销毁后所有模块引用置为 `null`，允许插件热重载后重新创建。
- `destroyPhone()` 只清理动态主题样式，不删除 manifest 加载的基础 `dist/index.css`。
- 页面卸载监听只绑定一次，并在热重载时解除旧监听。
- 插件禁用和热重载应等待异步 `destroyPhone()`
  完成；浏览器强制刷新/关闭无法保证异步完成，只能依赖隐藏前已经持久化的恢复日志在下次启动核对。
- 任何初始化异常统一捕获并记录；找不到主页面 `body/head` 时不留下半成品节点。

### 6.7 生成服务

- 全插件只有一个 `GenerationService`、一把来源隐藏互斥锁和一套任务状态机；任何 App 都不得复制
  `generate()`、流式监听、停止、楼层隐藏/恢复或聊天切换保护。
- 生成内核只暴露统一 `generateContent(adapter, config)` 接口。App 通过按 `appId + actionId`
  注册的适配器提供业务差异：配置校验、提示词区段、固定输出契约、解析和保存；“日记普通生成/阅读反应”“论坛首帖/续回”“番外新建/续写/章节总结”等分别是 action，不是新的生成服务。
- `request` 明确区分楼层选择、额外上下文、引用素材、App/类型提示词、用户要求、固定格式和输出模式。

```ts
interface GenerationAdapter<TConfig, TResult> {
  appId: AppId;
  actionId: string;
  configSchema: z.ZodType<TConfig>;
  buildRequest(config: TConfig): GenerationRequestParts;
  parse(raw: string, config: TConfig): ParseResult<TResult>;
  save(result: TResult, context: GenerationSaveContext): Promise<{ entityId: string }>;
}
```

- `buildRequest()` 只能返回声明式区段和固定格式，不能直接调用 Tavern Helper、改楼层或订阅全局事件。
- `GenerationService` 先校验 adapter/config，再执行统一 token 预检和来源隐藏事务；`generate()`
  返回后必须先恢复并核验楼层，之后才能调用 adapter 的 `parse()`、通用预览流程和
  `save()`。解析、预览和业务持久化不处于楼层隐藏窗口内。
- `resultMode: preview|save` 与 `stream`
  的四种组合由生成内核处理，App 不维护自己的第二套生成状态。Adapter 只决定结果 DTO 和保存位置。
- Adapter 抛出的校验、解析和保存错误统一转换为结构化错误；保存失败不能重新发送生成请求，原始输出进入该 App 的失败草稿。
- `generationRegistry` 注册时验证 `(appId, actionId)`
  唯一，并提供开发期契约测试：同一测试适配器必须跑通 tavern/external、stream/non-stream、preview/save、停止、失败和聊天切换矩阵。
- 生成前对“实际可见来源楼层 + 续写/引用内容 + 完整 user_input”执行 token 预检，并为当前预设、世界书和输出至少保留 20% 上下文余量；明确超过可用窗口时阻止生成并列出各区段占用，禁止静默删除用户选中的楼层、帖子、章节或书信。外部 API 优先使用其可选
  `contextWindow/maxOutputTokens`，留空时使用酒馆当前值并标记为估算。
- `sourceVisibility` 负责快照、恢复日志、临时隐藏、聊天一致性检查和 `finally` 回滚；生成服务不能自行零散修改楼层状态。
- `generateContent()` 必须在来源隐藏事务的回调内部调用 `generate()`；除来源选择外的内容按 3.6 的顺序进入 `user_input`。
- 不覆盖
  `chat_history`，让酒馆从临时可见的真实楼层构建聊天历史，以保留角色名、群聊身份、当前 swipe、正则和世界书深度行为。
- 不传预设、世界书文本或额外注入；当前预设和世界书完全由 `generate()` 使用酒馆当前环境处理。默认文本通道不传
  `custom_api`，外部文本通道只由 `textProvider` 附加已验证的 `custom_api`，其余请求结构完全相同。
- `externalApiClient`
  参考 Diary 的职责边界，只负责 URL 规范化、经 SillyTavern 后端测试连接、整理模型列表和产生脱敏错误；它不调用
  `/gen`，不监听或修改全局 `CHAT_COMPLETION_SETTINGS_READY`，也不直接负责内容生成。
- `textApiSettings`
  的队列保存必须返回可等待的真实 Promise；连接测试只有在模型列表成功解析并且配置文件保存成功后才显示“连接成功”，不能像 fire-and-forget 队列一样提前报告成功。
- 生成前读取当前预设状态；若 `chatHistory`、`worldInfoBefore` 或 `worldInfoAfter`
  等相关占位位置未启用，显示明确风险警告，但仍允许用户继续生成。
- 为每次任务生成唯一任务 ID，停止旧任务不能影响新任务。
- 流式监听只接收当前 `generation_id`，并确保结束后解除；停止操作只停止对应任务。
- 隐藏事务持有全局互斥锁；锁释放、`SendGuard` 清理和恢复日志清理只能发生在楼层状态成功还原之后。
- XML Parser 由各 App action adapter 注册，不在生成内核或一个超长函数里堆条件分支。
- 解析结果统一返回 `data + raw + warnings`。
- 失败使用结构化错误，不用“特殊字符串是否包含暂不可用”判断状态。

### 6.8 UI 组件

优先建立以下共享组件后再写 App：

- 手机壳、状态栏、标题栏、Home 条；
- App 图标、书架、列表、空状态；
- 搜索栏、筛选条、分段控制器；
- 阅读器工具栏；
- 确认对话框、提示弹窗；
- 生成配置、生成遮罩、流式输出、结果预览；
- 来源信息面板；

筛选条必须使用可测试的横向滚动容器，子项禁止收缩；纵向页面滚动和横向筛选滚动不得使用同一 overflow 容器。

### 6.9 保存内容的前端渲染

保存内容不在酒馆聊天楼层中，因此渲染链路不得依赖楼层 ID、当前聊天 DOM 或 Tavern Helper 的楼层显示格式化函数：

```text
持久层中的原始 HTML
  → buildFrontendDocument() 补齐完整文档与统一安全策略
  → FrontendFrame.vue 写入 iframe.srcdoc
  → iframe 内独立执行 HTML/CSS/JavaScript 并自行滚动
```

- `FrontendFrame` 只接收正文和渲染配置，不读取聊天记录，也不负责持久化。
- 参考 `st-theater-main` 的 `currentDisplayHtml`
  思路，复制、导出、续写和重新构建始终读取 Repository/Store 中的原始 HTML 快照，绝不读取
  `iframe.srcdoc`、`contentDocument` 或脚本运行后 DOM 作为持久数据。
- `buildFrontendDocument()` 始终以插件固定 CSP 覆盖正文自带的 CSP，并移除 `<base>`、`meta[http-equiv="refresh"]` 和嵌套
  `<iframe>/<object>/<embed>`；允许的 HTTPS 静态资源仍受 sandbox 隔离，正文无法通过脚本主动请求 API。
- `FrontendFrame` 记录首次 `srcdoc`
  load；之后若 iframe 发生第二次顶层 load，视为正文尝试自导航，立即卸载 iframe 并显示“已阻止页面跳转”。`navigate-to 'none'`
  作为第一层限制，父组件 load guard 作为浏览器不支持该指令时的兜底。
- iframe 使用 `srcdoc`，不需要 Blob URL；如果以后因资源体积改用 Blob URL，组件卸载时必须调用 `URL.revokeObjectURL()`。
- 组件应以 `key` 或显式重建方式切换内容，避免上一条内容的脚本、定时器和页面状态泄漏到下一条。
- 父子页面如需通信，只允许约定的 `postMessage` 消息，并同时校验 iframe 的
  `contentWindow`、消息类型和载荷结构；正文脚本不能直接访问手机 Store、SillyTavern API 或父页面 DOM。
- 第一版唯一允许的子到父消息为内部高度上报；高度按移动端和桌面端分别 clamp，并用
  `ResizeObserver`/节流降低频率。生成正文自行发送的其他消息全部忽略。
- Markdown 模式继续使用手机自己的 Markdown 阅读组件；只有明确选择前端渲染的内容才进入 iframe。
- Markdown 阅读器固定关闭原始 HTML，渲染结果再经过 DOMPurify；链接只允许 `http/https/mailto`，外部链接添加
  `target="_blank" rel="noopener noreferrer"`。搜索高亮通过文本节点实现，不把正文或查询字符串拼进 `innerHTML`。

## 7. 实现中禁止采用的问题模式

- 不保留接近千行的单体数据 Store；按领域和持久化职责拆分。
- 不在模板中堆大量行内样式。
- 不使用宽泛 `any` 处理选择框和导入数据。
- 不用字符串包含关系判断生成成功或失败。
- 不让 App 自己重复实现楼层解析、复制、详情导航和生成状态。
- 不把跨聊天快照和当前聊天实体混成同一种可编辑对象。
- 不保留八股筛选横向滚动缺陷。
- 不默认信任导入 JSON 或未知变量结构。
- 不新增与插件定位冲突的独立 AI、预设、世界书和生图配置。

## 8. 实现顺序

### 阶段 0：仓库基线

- 在当前插件仓库保留本文档、酒馆内验收清单和必要的数据样例；参考项目只放在 `docs/reference`。
- 冻结本插件的 zod Schema、六类 `<content>` 输出样例、默认提示词和默认八股规则。
- 明确只支持本插件自身 `schemaVersion` 升级，不承担不存在的旧实现迁移。

### 阶段 1：框架

- 基于当前仓库已有 manifest、Vite、TypeScript 和 ESLint 基线实现生命周期、Tavern
  Helper 依赖适配、扩展菜单启动入口和悬浮球。
- 手机壳、主题、主屏、App 注册表、App 模块契约、懒加载和导航栈。
- 设置 Repository 和基础错误处理。

### 阶段 2：数据与生成底座

- 先完成可注册 storage domain、zod schema、`user/files` 存储网关、settings 兜底、聊天 scope Repository、迁移和导入导出。
- 再完成唯一的
  `GenerationService`：楼层读取、来源隐藏事务、崩溃恢复日志、互斥锁、两种文本通道、流式状态、停止和聊天切换保护。
- 建立开发环境专用 `generationLab` adapter，不保存任何业务实体；先跑通 tavern/external × stream/non-stream ×
  preview/save，以及停止、异常、刷新恢复和切换聊天测试矩阵。
- 生成事务没有通过上述矩阵前，不开始复制任何创作 App 页面或业务生成代码。

## 9. 完成标准

插件只有同时满足以下条件才算实现完成：

- 12 个 App 均能从主屏进入且返回逻辑正确。
- 12 个 App 均通过独立 `PhoneAppDefinition` 注册并按需加载；删除或加载失败一个 App 模块不会阻止手机壳和其他 App 启动。
- 新增内部测试 App 时无需修改手机壳、导航、`GenerationService` 或 storage
  index 类型，只需注册 App 定义、路由及可选的 storage domain/generation
  adapter；未知 App 与 domain 数据在降级后不会被自动删除。
- Tavern Helper 在 `APP_READY` 后完成强制依赖检查；依赖未就绪时不会进入半初始化状态或执行生成、楼层修改和聊天归档读取。
- 所有创作 App 共用唯一 `GenerationService`、来源隐藏锁和任务状态机；仓库中不存在 App 私有的 `generate()`
  调用、楼层恢复实现或全局流式事件订阅。
- 开发期 generationLab 已覆盖 tavern/external × stream/non-stream ×
  preview/save，以及停止、API 失败、解析失败、保存失败、聊天切换和刷新恢复矩阵。
- 6 个创作 App 均通过“读取楼层 → 临时隐藏来源外楼层 → generate() → 恢复隐藏状态 → 解析 → 预览/保存 → 详情”实测。
- 最新、全部、单层、最近 N 楼和自定义范围均只让选中的原本可见楼层参与生成；生成完成后所有楼层隐藏状态与生成前完全一致。
- 来源范围没有任何可见楼层时阻止生成；恢复期间发生新增、删除或楼层身份哈希变化时不会错误写回其他楼层。
- App/类型提示词、引用素材、额外要求和固定输出格式按约定顺序进入 `user_input`，固定格式不可编辑。
- 番外续写、论坛回复、书信回信和日记阅读反应均强制重新选择来源楼层，并分别携带已约定的章节、帖子、最近 N 封信或当前日记上下文；番外总结只替换其已覆盖的旧章节。
- 当前预设、群聊发言者、酒馆正则、世界书关键词激活和世界书深度条目在临时隐藏来源外楼层后仍能正常生效。
- 正常完成、用户停止、API 报错、解析报错、聊天切换和页面刷新场景均经过恢复测试，不会永久改变楼层隐藏状态或恢复到错误聊天。
- 原本隐藏的楼层恢复后仍隐藏；两个并发手机任务和酒馆正在生成时不能同时进入来源隐藏事务。
- 来源隐藏期间酒馆 UI 发送会被 `SendGuard`
  拦截；事务结束不调用全局按钮启用/禁用 API，其他扩展开始生成时手机任务会停止并先恢复楼层。
- 当前预设关闭相关占位位置时会警告，用户选择继续后仍可生成。
- `resultMode: preview|save` 与 `stream: boolean` 的四种组合行为明确且均通过测试。
- 并发或连续生成时，流式内容和停止操作不会串到其他 `generation_id`。
- 解析失败可查看原文并重新解析。
- 当前聊天数据持久化，切换聊天不串数据。
- Branch/Checkpoint 自动获得新的 `scopeId` 并复制原手机数据，聊天改名沿用原
  `scopeId`；无法判定时由用户选择，不自动串库。
- 阅读聊天能列出当前角色卡全部聊天，只展示 AI 输出，并分别按用户的标题/正文正则生成阅读内容；隐藏 AI 楼层默认不显示且可手动开启，切换角色卡后不沿用上一角色缓存。
- 阅读聊天按实测字段正确解析 `mes`、`is_user`、`is_system`、`swipe_id` 和
  `swipes`；目录首屏不为统计 AI 楼层数而批量下载全部聊天全文。
- 阅读详情离开后不保留长期正文缓存，重新打开和手动刷新能读到同长度编辑后的新内容。
- 恶意或灾难性回溯正则在 Worker 超时后被终止，不会卡死 SillyTavern 主线程；超大替换结果不会进入 UI。
- 小剧场前端渲染直接把持久层中的原始 HTML 构造成完整 `srcdoc`
  并在沙盒 iframe 中显示，不创建或依赖聊天楼层；切换详情或关闭手机后能正确销毁 iframe，重新打开时按原导航状态重建，且不持久化 iframe
  DOM 和运行状态。
- 小剧场 iframe 能加载允许的 HTTPS 静态资源，但
  `fetch`、XHR、WebSocket、EventSource、嵌套页面、表单外发、顶层导航和父页面访问均被固定 sandbox/CSP 阻止，正文自带标签不能放宽策略。
- 小剧场在生成前选择 Markdown 或前端渲染模式，两种结果都从最外层 `<content>`
  原样提取；HTML 模式不要求 CDATA，也不会被当作 Markdown 二次格式化。
- Markdown 模式中的原始 HTML、危险 URL 和搜索文本不能形成主页面 XSS；只有显式 frontend 模式能够在沙盒 iframe 中执行脚本。
- 每个 `scopeId` 的日记、论坛、番外、小剧场、书信和总结分别使用六个逻辑文件及不可变 revision，并能在 SillyTavern
  `user/files`
  正常创建、读取和提交；修改一类内容不会重写其他 scope 或类型，索引提交失败仍使用旧 revision，浏览器数据清理后仍可恢复全部内容和设置。
- settings 兜底只保留最近活动 scope，且总量不超过
  `5 MiB`；超限文件会明确提示未进入兜底，不会让 settings 再次累积成全部聊天的单体仓库。
- 除壁纸原文件外，JSON revision 不存在、损坏、读取失败和写入失败时会切换到 `extension_settings.sillytavernPhone`
  兜底并显示来源状态；storage
  index 损坏时能由不受容量淘汰的 index 镜像恢复。主文件与兜底均失败时明确报错。壁纸原文件失败时只保留索引前状态并明确提示，不把图片写入 settings。
- 正常关闭手机会刷新本次变更领域的 settings 兜底；导入、迁移和清空后会立即刷新，强制关闭时允许兜底落后一个会话。
- 聊天改名后仍关联原 `scopeId`；单角色使用头像文件 ID、群聊使用 groupId 形成稳定
  `ownerKey`，同名角色、不同聊天和群聊不会共用创作数据。
- 删除壁纸或用户确认清理未绑定 scope 时，实体文件通过 `/api/files/delete` 删除；部分失败可重试且不会提前丢失 storage
  index。
- 跨聊天收藏能显示只读快照。
- 收藏原文编辑后跨聊天快照同步更新；取消收藏、删除原文或删除分册后快照消失，跨文件写入失败时能通过
  `pendingFavoriteSyncs` 自动重试且不会显示待删除快照。
- 完整 ZIP 备份导入会在全量预检和二次确认后写入新 revision，并以一次 storage
  index 提交替换全部插件数据；选择性 JSON 导入只合并到当前 scope、为全部实体生成新 ID 并正确重映射内部引用。
- 导出、导入、清空、本插件版本迁移和损坏文件恢复经过实测。
- 所有事件监听、计时器和插件主页面节点均能正确销毁。
- 插件不创建 SillyTavern 扩展设置面板；除最小启动入口外，全部设置均在手机设置 App 内完成。
- 重复初始化、热重载和多次点击入口不会重复创建扩展菜单入口、手机根节点、悬浮球、样式节点或事件监听。
- 悬浮球在鼠标、触摸和触控笔下均能区分点击与拖拽，位置可恢复，窗口尺寸变化后不会留在屏幕外。
- 八股筛选在窄屏可横向滑动。
- 手机内不存在独立角色卡、预设、世界书或生图入口；外部文本 API 只作为可选生成通道，通过 Tavern Helper
  `generate({ custom_api })` 使用，不能绕过当前预设、世界书和统一来源楼层管线。
- 酒馆当前 API 与外部文本 API 都通过完整生成管线测试；外部 API 的 URL 规范化、空 Key、本地代理、模型刷新、连接失败、配置改变后失效、密钥遮罩、切回酒馆通道和不导出密钥的默认行为均符合约定。
- lint 为 0 error / 0 warning，开发与生产构建均成功。
- 完成酒馆内验收清单，没有阻断级问题。
